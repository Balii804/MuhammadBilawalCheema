# GGI Backend Test - AI Chat & Subscription System

A TypeScript-based backend application implementing an AI Chat module and Subscription Bundle module following REST principles, Domain-Driven Design (DDD), and Clean Architecture.

## Architecture

This project follows **Clean Architecture** with **Domain-Driven Design** principles:

- **Domain Layer**: Core business logic, entities, and domain services
- **Application Layer**: DTOs, controllers, and route handlers
- **Infrastructure Layer**: Database repositories (Prisma), external service clients
- **Shared Layer**: Common utilities, error handling, database client

## Features

### Module 1: AI Chat Module

- ✅ Accepts user questions and returns mocked OpenAI responses
- ✅ Stores questions, answers, and tokens in the database
- ✅ Tracks monthly usage per user:
  - Each user gets **3 free messages per month**
  - Free quota automatically resets on the 1st of each month
  - After free quota, requires a valid subscription bundle
- ✅ Supports multiple active bundles per user with different tiers:
  - **Basic**: 10 responses
  - **Pro**: 100 responses
  - **Enterprise**: Unlimited responses
- ✅ Deducts usage from the bundle with the latest remaining quota
- ✅ Throws structured errors when user exceeds quota
- ✅ Simulates OpenAI API response time delay (500-1500ms)

### Module 2: Subscription Bundle Module

- ✅ Allows users to create subscription bundles (Basic, Pro, Enterprise)
- ✅ Supports billing cycles: Monthly or Yearly
- ✅ Toggle auto-renew functionality
- ✅ Each subscription includes: maxMessages, price, startDate, endDate, renewalDate
- ✅ Simulates billing logic:
  - Auto-renew subscriptions if enabled
  - Random payment failure simulation (20% failure rate)
  - Marks subscription inactive on payment failure
- ✅ Supports cancellation:
  - Ends current billing cycle
  - Prevents renewal
  - Preserves usage history

## Setup Instructions

### Prerequisites

- Node.js (v18 or higher)
- npm or yarn or pnpm

### Installation

1. **Clone the repository** (or navigate to the project directory)

2. **Install dependencies**:
   ```bash
   npm install
   ```

3. **Set up environment variables**:
   ```bash
   cp .env.example .env
   ```
   Edit `.env` if needed (default SQLite database is fine for development)

4. **Set up the database**:
   ```bash
   npx prisma generate
   npx prisma migrate dev --name init
   ```

5. **Start the development server**:
   ```bash
   npm run dev
   ```

   The server will start on `http://localhost:3000` (or the port specified in `.env`)

## API Documentation

### Base URL

```
http://localhost:3000/api
```

### Chat Endpoints

#### POST `/api/chat`

Send a chat message/question.

**Request Body**:
```json
{
  "userId": "user123",
  "question": "What is TypeScript?"
}
```

**Response** (200 OK):
```json
{
  "question": "What is TypeScript?",
  "answer": "This is a mocked OpenAI response to your question: \"What is TypeScript?\". In a real implementation, this would be generated by GPT models.",
  "tokensUsed": 45,
  "timestamp": "2024-01-15T10:30:00.000Z",
  "remainingFreeMessages": 2,
  "usedQuotaType": "FREE",
  "subscriptionId": null
}
```

**Error Response** (402 Payment Required):
```json
{
  "error": {
    "code": "INSUFFICIENT_QUOTA",
    "message": "User has exceeded available message quota.",
    "details": "You have exhausted your free monthly quota and do not have an active subscription bundle. Please subscribe to continue using the service."
  }
}
```

#### GET `/api/chat/history?userId=user123`

Get chat history for a user.

**Query Parameters**:
- `userId` (required): User ID
- `limit` (optional): Number of messages to return (default: 50)

**Response** (200 OK):
```json
{
  "messages": [
    {
      "id": "msg123",
      "question": "What is TypeScript?",
      "answer": "This is a mocked OpenAI response...",
      "tokensUsed": 45,
      "createdAt": "2024-01-15T10:30:00.000Z"
    }
  ]
}
```

### Subscription Endpoints

#### POST `/api/subscriptions`

Create a new subscription bundle.

**Request Body**:
```json
{
  "userId": "user123",
  "tier": "PRO",
  "billingCycle": "MONTHLY",
  "autoRenew": true
}
```

**Tier Options**: `BASIC`, `PRO`, `ENTERPRISE`
**Billing Cycle Options**: `MONTHLY`, `YEARLY`

**Response** (201 Created):
```json
{
  "id": "sub123",
  "userId": "user123",
  "tier": "PRO",
  "billingCycle": "MONTHLY",
  "maxMessages": 100,
  "messagesUsed": 0,
  "price": 50,
  "startDate": "2024-01-15T10:30:00.000Z",
  "endDate": "2024-02-15T10:30:00.000Z",
  "renewalDate": "2024-02-15T10:30:00.000Z",
  "autoRenew": true,
  "status": "ACTIVE",
  "createdAt": "2024-01-15T10:30:00.000Z",
  "updatedAt": "2024-01-15T10:30:00.000Z"
}
```

```json
      "maxMessages": 100,

      "endDate": "2024-02-15T10:30:00.000Z",
      "renewalDate": "2024-02-15T10:30:00.000Z",
      "autoRenew": true,
      "status": "ACTIVE",
      "createdAt": "2024-01-15T10:30:00.000Z",
      "updatedAt": "2024-01-15T10:30:00.000Z"
    }

```

#### POST `/api/subscriptions/:id/cancel`

Cancel a subscription (disables auto-renew, remains active until end date).

**Response** (200 OK):
```json
{
  "message": "Subscription cancelled successfully. It will remain active until the end of the billing cycle.",
  "subscription": {
    "id": "sub123",
    "autoRenew": false,
    ...
  }
}
```

#### POST `/api/subscriptions/:id/renew`

Manually trigger renewal for a subscription (for testing/demo purposes).

**Response** (200 OK - Success):
```json
{
  "success": true,
  "message": "Subscription renewed successfully",
  "newBundle": {
    "id": "sub123",
    "messagesUsed": 0,
    ...
  }
}
```

**Response** (400 Bad Request - Failure):
```json
{
  "success": false,
  "message": "Renewal failed. Payment may have failed or subscription is not eligible for renewal."
}
```

## Usage Examples

### Example 1: Using Free Quota

1. Send 3 messages (all free):
   ```bash
   curl -X POST http://localhost:3000/api/chat \
     -H "Content-Type: application/json" \
     -d '{"userId": "user1", "question": "Hello"}'
   ```

2. 4th message will fail (quota exceeded):
   ```bash
   curl -X POST http://localhost:3000/api/chat \
     -H "Content-Type: application/json" \
     -d '{"userId": "user1", "question": "Another question"}'
   ```

### Example 2: Creating Subscription and Using It

1. Create a PRO subscription:
   ```bash
   curl -X POST http://localhost:3000/api/subscriptions \
     -H "Content-Type: application/json" \
     -d '{
       "userId": "user1",
       "tier": "PRO",
       "billingCycle": "MONTHLY",
       "autoRenew": true
     }'
   ```

2. Now user can send messages beyond free quota (up to 100 for PRO tier)

3. Check subscription status:
   ```bash
   curl "http://localhost:3000/api/subscriptions?userId=user1"
   ```

## Error Handling

All errors follow a structured format:

```json
{
  "error": {
    "code": "ERROR_CODE",
    "message": "Human-readable error message",
    "details": "Additional details (optional)"
  }
}
```

**Error Codes**:
- `INSUFFICIENT_QUOTA` (402): User has exceeded available message quota
- `NOT_FOUND` (404): Resource not found
- `VALIDATION_ERROR` (422): Invalid request data
- `INTERNAL_ERROR` (500): Server error

## Database Schema

The application uses Prisma with SQLite (easily switchable to PostgreSQL). Key models:

- **User**: Basic user information
- **ChatMessage**: Stores all chat interactions
- **UserMonthlyUsage**: Tracks free message usage per month
- **SubscriptionBundle**: Subscription bundles with tier, billing cycle, usage tracking

## Design Decisions

1. **Clean Architecture**: Separates domain logic from infrastructure, making the codebase testable and maintainable.

2. **DDD Principles**: Each module (chat, subscriptions) is a bounded context with its own domain models and services.

3. **Repository Pattern**: Abstracts data access, allowing easy swapping of database implementations.

4. **Structured Errors**: Domain errors are mapped to HTTP status codes consistently.

5. **Monthly Quota Reset**: Implemented by using month keys (YYYY-MM), automatically giving users fresh quota each month.

6. **Bundle Selection**: When multiple active bundles exist, the system selects the one with the latest end date (most remaining time).

7. **Payment Simulation**: 20% random failure rate to simulate real-world payment processing.

## Notes

- The OpenAI integration is mocked with simulated delays (500-1500ms)
- Free quota resets automatically on the 1st of each month (tracked by month key)
- Subscription renewals can be triggered manually via the `/renew` endpoint
- Usage history is preserved when subscriptions are cancelled
- Enterprise tier has unlimited messages (`maxMessages: null`)
